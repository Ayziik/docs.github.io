01

Problem Statement

Контекст, симптоми, наслідки та кореневі причини

🔥 Симптоми

  • Сотні розрізнених задач без спільного контексту
  • Неможливо побачити прогрес по угоді як ціле
  • Файли розпорошені між задачами і коментарями
  • Менеджери дізнаються про затримки постфактум

⚡ Бізнес-наслідки

  • ~10–15 хв на пошук одного файлу по угоді
  • Пропущені задачі → зриви дедлайнів по угодах
  • Надмірне ручне планування з боку керівника
  • Нова людина не може швидко увійти в контекст

🎯 Хто страждає найбільше

  • Юридичні фірми — угода = 50+ задач і документів
  • Виробничі підприємства — паралельні процеси
  • Агентства — окремі треки для кожного клієнта
  • Будь-яка команда 5+ людей зі спільними угодами

🔍 Кореневі причини

  • Відсутня групуюча сутність між CRM-об'єктом і задачею
  • Немає спільного статусного workflow для групи задач
  • Немає єдиного файлового простору для процесу
  • Немає шаблонування — кожен процес вручну
ℹ️
Проблема не в задачах як таких — задачі працюють. Проблема у відсутності структури вище рівня задачі, що об'єднує роботу навколо спільної мети або угоди.
02

Візія

Що і навіщо ми будуємо

Дати командам єдиний простір для управління роботою навколо конкретної мети — зберігаючи простоту для тих, кому проєктне управління не потрібне.

📐

Прогресивне ускладнення

Базова система задач залишається незмінною. Проєкти — опціональний шар, що вмикається однією налаштункою.

🗂️

Контекст завжди поряд

Файли, задачі, прогрес і команда — в одному місці, без перемикання між вкладками.

⚙️

Гнучкість процесу

Кожна команда налаштовує власні статуси та шаблони. Система адаптується до процесу, а не навпаки.

🔒

Безпека даних

Подвійна система прав: хто бачить проєкт і хто бачить конкретні задачі всередині.

Скоуп

Що входить і що не входить у версію 1.0

✅ In Scope — v1.0

  • Сутність «Проєкт» з повним CRUD
  • Ієрархія: Проєкт → Задача → Підзадача → Чек-лист
  • Два типи задач: Standalone та Проєктна
  • Кастомні статуси задач у проєкті
  • Зв'язок проєкту з CRM-сутністю (Угода)
  • Файловий дровер з пошуком і шерингом
  • Шаблони проєктів із відносними дедлайнами
  • Подвійна система прав доступу
  • Розділ «Моя Робота» (канбан за датами)
  • Lifecycle: архівування та відновлення
  • Прогрес-бар на рівні проєкту
  • Активація через налаштування (toggle)

🚫 Out of Scope — v1.0

  • Діаграма Ганта / часова шкала
  • Залежності між задачами
  • Бюджетування та облік ресурсів
  • Призначення команди при авторозгортанні шаблону
  • Звіти та аналітичні дашборди по проєктах
  • Мобільний нативний інтерфейс
  • Інтеграції з Jira / Asana / Trello
  • Нотифікації у Telegram / Slack
  • Time tracking у задачах
  • Множинне призначення відповідального
  • Вкладені підпроєкти
  • Зовнішній API для проєктів

07

Ключові сценарії

Як система виглядає в реальному контексті роботи

S1

Запуск проєкту з шаблону по угоді

🏆 Керівник
1

Угода переходить на стадію «В роботі»

Менеджер відкриває картку угоди. Тригер автоматизації спрацьовує або натискає «Створити проєкт» вручну.

2

Вибір шаблону

З'являється список шаблонів. Вибирає «Юридичний супровід угоди».

⚡ Система розгортає 12 задач, 3 статуси, відносні дедлайни
3

Налаштування команди

Додає юриста та асистента. Призначає відповідальних по конкретних задачах.

4

Проєкт активовано

Учасники бачать задачі в «Моя Робота». Файли угоди доступні у файловому дровері.

✓ Прогрес-бар 0/12, дедлайни розраховані з урахуванням вихідних
S2

Ранковий огляд задач

👷 Виконавець
1

Відкриває «Моя Робота»

Розділ завжди доступний незалежно від модуля проєктів. Канбан-дошка за датами.

2

Сканує «Протерміновані» і «Сьогодні»

На картках проєктних задач — бейдж з назвою проєкту. Standalone задачі без бейджа.

3

Відкриває задачу, змінює статус

Переміщує картку на канбані. Прогрес-бар проєкту оновлюється автоматично.

⚡ Прогрес: 4/12 → 5/12 задач виконано
S3

Перевірка стану проєкту

🏆 Керівник
1

Відкриває список проєктів

Бачить усі активні проєкти з прогрес-барами та дедлайнами на одному екрані.

2

Відкриває картку проєкту

Хедер: аналітика по задачах, терміни, команда. Канбан показує розподіл задач по статусах.

3

Знаходить файл і шерить клієнту

Відкриває файловий дровер, шукає за назвою, копіює посилання — без виходу з контексту.

✓ Час пошуку файлу: менше 1 хвилини
08

Функціональні блоки

Опис ключових модулів системи

📁

Проєкт

Core
  • ◆Обов'язкові поля: Назва, Керівник
  • ◆Опціональні: Опис, Файли, Термін виконання, Команда, Додаткові статуси
  • ◆Зв'язок з CRM: якщо проєкт створюється з картки Угоди — автоматично успадковує зв'язок. При видаленні CRM-сутності зв'язок зникає, проєкт залишається
  • ◆Картка проєкту — один екран: Хедер (overview + аналітика) + Канбан задач + Файловий дровер
  • ◆Прогрес-бар: закриті задачі / загальна кількість. Підзадачі не враховуються
✅

Задачі та ієрархія

Core
  • ◆Standalone задача: існує поза проєктами. Стани «Відкрито» / «Закрито». Всі існуючі задачі після оновлення стають цим типом автоматично
  • ◆Проєктна задача: прив'язана до проєкту, рухається по кастомних статусах. Зв'язок із проєктом незмінний після створення
  • ◆Підзадача: задача де parent — інша задача. Має власний чек-лист
  • ◆Чек-лист: вбудований інструмент всередині задачі або підзадачі, не окрема ієрархічна сутність
📋

Шаблони проєктів

Config
  • ◆Вміст: статуси, задачі, підзадачі, чек-листи, описи, теги. Без персональних даних
  • ◆Дедлайни: відносні (Дата старту + N робочих днів) з урахуванням вихідних
  • ◆Ручне розгортання: «Створити з шаблону» у розділі Проєктів
  • ◆Авторозгортання: шаблон прив'язується до CRM-тригера (напр. «Угода → стадія X»)
  • ◆Зберегти як шаблон: з налаштувань активного проєкту. Структура копіюється, персональні дані очищаються
🗂️

Розділ «Моя Робота»

UI
  • ◆Доступний завжди: незалежно від увімкнення модуля проєктів
  • ◆Канбан за датами: Протерміновані · Сьогодні · Завтра · Цей тиждень · Наступний тиждень · Майбутні · Виконано
  • ◆Змішаний контент: Standalone і Проєктні задачі на одній дошці. На проєктній — бейдж із назвою проєкту
  • ◆Перемикачі: «Канбан / Список» + (для адмінів) «Мої задачі / Всі задачі» з фільтрами по співробітниках
  • ◆Глобальний пошук: по всіх задачах, включно з проєктними

Ризики та допущення

Що може піти не так і як це митигувати

Ризик Рівень Стратегія митигації
Scope creep — «додайте Ганту» High Чіткий Out of Scope документ. Окремий backlog для v2. Рев'ю скоупу на кожному sprint planning.
Низький adoption — команди не переходять High Ранній Beta з цільовими клієнтами. Онбординг-флоу. Відстеження метрик з дня релізу.
Міграція даних — збій конвертації задач Med Dry run міграції на staging. Rollback план. Логування всіх конвертацій.
Продуктивність при великих проєктах (500+ задач) Med Load testing на Alpha. Pagination/virtualization у списках. Soft limit з попередженням.
Складність прав — команди плутаються Med Usability testing прав у Beta. Чіткі UI-підказки при обмеженнях. Документація для адмінів.
Видалення проєкту через помилку Low Двоетапне підтвердження. Ввід назви проєкту для підтвердження. Тільки Адмін.

Ключові допущення

📐 Технічні

  • Існуюча схема задач підтримує розширення без повної міграції
  • CRM API повертає стабільні ID для зв'язків
  • Файловий сервіс підтримує агрегацію по entity_id

📊 Продуктові

  • Цільові клієнти готові брати участь у бета-тестуванні
  • Більшість команд потребує ≤ 10 кастомних статусів
  • Шаблони дадуть достатній ROI для команд з 3+ проєктами/місяць
